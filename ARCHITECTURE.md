# ARCHITECTURE.md — System Design & Integrations

> Companion to [`README.md`](./README.md). This document explains the **why** behind the codebase structure: how the dual-mode runtime works, how dependencies are scoped, why both Firebase and Supabase exist, and how state and navigation flow.

---

## 1. Architectural Style — Feature-Driven Clean(-ish) Architecture

The codebase is **feature-driven on top of a Clean-Architecture-inspired layering**, but it is pragmatic rather than dogmatic: there is no separate `data/domain/presentation` triad inside every feature. Instead, the layering is enforced at the **module boundary**:

| Layer | Where it lives | What it contains |
| --- | --- | --- |
| **Presentation** | `lib/customer/features/<feature>/{screens,widgets}` and `lib/seller/features/<feature>/{screens,widgets}` | `MaterialApp` shells, screens, widgets. |
| **Application** | `lib/<mode>/features/<feature>/{bloc,cubit}` and `lib/<mode>/services/` | BLoCs / Cubits, long-lived services (e.g. `OrderTrackingService`, `NewOrdersListener`). |
| **Domain** | `lib/shared/models/` | Pure Dart entities (`Product`, `Order`, `Cart`, `Me`, …) shared by both modes. |
| **Data / Repository** | `lib/shared/repositories/` (+ feature-local data sources, e.g. `seller/features/products/data/`) | Abstract repositories + Mock / Remote (Dio) / Supabase / Hive / Hybrid implementations. |
| **Infrastructure** | `lib/core/` | DI container, network clients, theming, i18n, notifications, storage, error mapping. |

### Top-level `lib/` layout

```
lib/
├── main.dart                  # bootstrap, Firebase init, Phoenix-wrapped mode router
├── firebase_options.dart      # generated by flutterfire (Android + iOS only)
├── config/
│   ├── app_config.dart        # --dart-define gateway (env-driven constants)
│   └── app_mode.dart          # AppMode enum + AppModeCubit persistence
├── core/                      # CROSS-MODE infra (root DI scope)
│   ├── auth/                  # AuthCubit, AuthRepository, AppModeCubit, SignOut
│   ├── connectivity/          # ConnectivityService + NetworkCubit
│   ├── deep_links/            # DeepLinkService
│   ├── di/service_locator.dart# get_it scope wiring (root + mode)
│   ├── error/failure.dart     # sealed Failure hierarchy
│   ├── i18n/                  # Custom in-Dart translation system
│   │   ├── app_translations.dart
│   │   ├── app_locale_controller.dart
│   │   └── translations/      # 19 per-domain Dart maps (uz/ru/en)
│   ├── logging/               # talker + nav observers
│   ├── maps/                  # YandexMapKitInitializer (platform channel)
│   ├── network/               # Dio client, error_handler, supabase_client
│   ├── notifications/         # PushService, NotificationHandler, NotificationModel
│   ├── storage/               # HiveBoxes enum, SecureStorage, CacheStore
│   ├── theme/                 # AppTheme, SellerTheme, AppColors, AppFonts, ThemeCubit

│   ├── mock/                  # canned data + mock repositories
│   ├── utils/                 # image_upload helpers
│   └── widgets/               # cross-mode UI atoms (EmptyState, Shimmer, etc.)

    ├── seller_app.dart        # MaterialApp + onGenerateRoute

```

### Folder-naming conventions

- Each feature owns: `bloc/` *or* `cubit/`, `screens/`, optionally `widgets/`, optionally `data/`.
- Models, repository interfaces, mocks, and broadly-reused widgets live in `lib/shared/` so both modes can import without a circular `customer → seller` dependency.
- `lib/core/` never imports from `lib/customer/`, `lib/seller/`, `lib/auth/`, or `lib/shared/` (except very narrow indirect references — see refactor notes).

## 2. Dual-Mode Runtime — Phoenix + Scoped DI

The single most distinctive architectural feature is **runtime mode switching** without a process restart.

### Bootstrap sequence (`lib/main.dart`)

```
WidgetsFlutterBinding.ensureInitialized()
    │
    ▼
SystemChrome.setSystemUIOverlayStyle(...)   ← dark icons on light splash
    │
    ▼
Firebase.initializeApp(DefaultFirebaseOptions.currentPlatform)
FirebaseMessaging.onBackgroundMessage(firebaseMessagingBackgroundHandler)
    │
    ▼
initRootScope()                              ← Hive, Supabase, Dio, repos, root cubits
    │
    ▼
initModeScope(getInitialMode())              ← per-mode BLoCs, listeners, services
    │
    ▼
PushService.bootstrap()                      ← FCM foreground listener, token refresh
_wireAuthToPushTokens()                      ← AuthCubit → save FCM token after sign-in
_wirePushToInboxRefresh()                    ← FCM payload → NotificationsCubit.load()
    │
    ▼
AppLocaleController.fromBox(settingsBox)
AppTranslations.setInstance(forLocale(...))
    │
    ▼
runApp(MultiBlocProvider(
  ThemeCubit / AuthCubit / AppModeCubit,
  child: _AppRoot,                            ← Phoenix-wrapped
))
```

### The mode switch flow

```
button taps cubit.switchMode(seller)
                │
                ▼
AppModeCubit emits AppMode.seller (persisted to Hive 'app_mode')
                │
                ▼
BlocListener inside Phoenix:
  if (sl.currentScopeName == mode.name) return;   ← idempotency guard
  await sl.popScope();                            ← dispose customer mode singletons
  await initModeScope(mode);                      ← register seller mode singletons
  Phoenix.rebirth(context);                       ← throw away widget tree, rebuild
                │
                ▼
_ModeRouter rebuilds under a fresh key:
  switch (getInitialMode()) {
    AppMode.customer => CustomerApp(),
    AppMode.seller   => SellerApp(),
  }
```

### Why Phoenix instead of `Navigator.pushReplacement`?

- The two modes are **independent `MaterialApp`s** with different themes, different routers, and different BLoC trees. Phoenix gives us a clean unmount/remount cycle without re-running `main()`, which means Firebase, Supabase, and Hive stay initialised across mode flips.
- The DI scope swap happens **before** `Phoenix.rebirth`, so by the time the new tree mounts, `sl<SomeCubit>()` already resolves to the new mode's instance.

### Cold-start splash dwell

`_ModeRouter` enforces a 1400 ms minimum splash duration (`_minSplashDuration`) and a 360 ms crossfade (`_crossfadeDuration`) so:

1. On a fast device, brand splash is still visible long enough to register.
2. On mode switches, the same splash masks any visual flicker from the DI swap.

## 3. Dependency Injection — `get_it` with Named Scopes

DI is centralised in `lib/core/di/service_locator.dart` (~556 lines). The container uses **two named scopes**:

### Root scope — `initRootScope()`

Singletons that survive mode switches:

- All Hive boxes (named: `settings`, `cache`, `pendingRoute`, `onboardingDraft`, `favorites`, `cart`, `newsReads`).
- `SupabaseClient` (only if `AppConfig.hasSupabase`).
- `Dio` (lazy singleton, disposed via `dio.close(force: true)`).
- `ThemeCubit`, `AppModeCubit`, `AuthCubit`, `ConnectivityService`, `NetworkCubit`.
- `PushService`, `NotificationHandler`, `DeepLinkService`, `CacheStore`, `SecureStorage`.
- All shared repositories (`ProductRepository`, `CategoryRepository`, `ShopRepository`, `CartRepository`, `FavoritesRepository`, `OrderRepository`, `AddressRepository`, `RegionRepository`, `BannerRepository`, `NotificationsCubit`, …).
- All shared data sources (`SupabaseProductDataSource`, `CategoryDataSource`, `NotificationDataSource`, `NewsDataSource`).

### Mode scope — `initModeScope(mode)`

Singletons that exist only while a given mode is active:

| Mode | Registrations |
| --- | --- |
| `customer` | `HomeBloc` (auto-loads), `CartBloc` (auto-loads), `FavoritesBloc` (auto-fetches), `CategoriesBloc` (auto-fetches), `ProfileCubit`, `ProfileOrdersCubit`, `OrderTrackingService` |
| `seller` | `SellerDashboardCubit` (registered as **factory**, not singleton), `NewOrdersListener` |

### Registration patterns

```dart
// Eager singleton with auto-dispose
sl.registerSingleton<ThemeCubit>(
  ThemeCubit(boxes.settings),
  dispose: (c) => c.close(),
);

// Lazy singleton (built on first lookup)
sl.registerLazySingleton<Dio>(
  buildDioClient,
  dispose: (dio) => dio.close(force: true),
);

// Factory (new instance per resolution — used for short-lived cubits)
sl.registerFactory<SellerDashboardCubit>(
  () => SellerDashboardCubit(sl<SellerDashboardRepository>()),
);

// Named instance (used for the 7 Hive boxes)
sl.registerSingleton<Box>(boxes.settings, instanceName: HiveBoxes.settings);
final box = sl<Box>(instanceName: HiveBoxes.settings);
```

### Mock vs live selection

`AppConfig.useMocks` is a `bool.fromEnvironment('USE_MOCKS', defaultValue: true)`. The DI wiring branches on it:

- When `useMocks == true` → `MockXRepository` instances (for catalog-style endpoints).
- When `useMocks == false` → `RemoteXRepository(dio)` or `SupabaseXRepository(supabase)`.
- Even with `useMocks == true`, some repositories *prefer* Supabase if a client is available (e.g. banners, cart, favorites, orders). The matrix lives in `service_locator.dart` lines 267–460.

### Cross-scope access guard

```dart
if (sl.isRegistered<NotificationsCubit>()) {
  sl<NotificationsCubit>().load();
}
```

Use this whenever a callback may fire from a scope where a cubit is not registered (e.g. push tap firing while seller mode is active but listener is wired to a customer-scoped cubit).

## 4. Backend — Why Both Firebase AND Supabase?

This dual-backend setup is intentional. Each provider handles only what it is best at:

### Supabase — the data plane

Used for **everything that needs a server-side database, auth, or live updates**.

| Supabase product | Used for | File pointers |
| --- | --- | --- |
| Postgres | All domain data: `products`, `categories`, `orders`, `carts`, `favorites`, `notifications`, `device_tokens`, etc. | `shared/repositories/supabase_*.dart` |
| Auth (email + password) | Single source of truth for user identity. Same session in both modes. | `core/auth/auth_repository.dart`, `auth/login_screen.dart`, `auth/register_screen.dart` |
| Realtime (Postgres CDC) | Order status updates (customer side), new orders (seller side), notification inbox. | `customer/services/order_tracking_service.dart`, `seller/services/new_orders_listener.dart`, `shared/repositories/supabase_notifications_repository.dart` |
| Storage | Product images, passport KYC uploads. | `shared/utils/image_upload.dart`, `seller/features/verification/` |
| Edge Functions | `API_BASE_URL` points at `<project>.supabase.co/functions/v1` — invoked via Dio for endpoints not modelled directly as tables. | `core/network/api_client.dart` |
| RLS | Enforces per-user row visibility (auth.uid()). Critical because the anon key is shipped to the client. | All Supabase repositories rely on this. |

Initialised in `core/network/supabase_client.dart`:

```dart
Future<SupabaseClient?> initSupabase() async {
  if (!AppConfig.hasSupabase) return null;
  await Supabase.initialize(url: AppConfig.supabaseUrl, anonKey: AppConfig.supabaseAnonKey);
  return Supabase.instance.client;
}
```

### Firebase — push only

Used **strictly for FCM** (push notifications). Firestore, Firebase Auth, and Firebase Storage are *not* in use.

| Firebase product | Used for | File pointers |
| --- | --- | --- |
| FCM | Topic broadcasts (`news`) + per-device personal pings. Token persisted to Supabase `device_tokens` table. | `core/notifications/push_service.dart`, `core/notifications/notification_handler.dart` |
| `flutter_local_notifications` | Foreground display of FCM payloads (Android only auto-shows when backgrounded). | `core/notifications/push_service.dart` |

Configured via `firebase_options.dart` (generated by `flutterfire configure`) — both `android` and `ios` `FirebaseOptions` are populated, with `projectId: woody-b3c1a`.

### Push flow (cross-mode aware)

```
FCM tap (cold start / background / foreground)
        │
        ▼
PushService._onMessageTapped(message)
        │
        ├── extract { route, mode, kind } from data payload
        │
        ▼
NotificationHandler.savePendingRoute(route, mode, ts)
        │
        ▼
if (mode != currentMode) → AppModeCubit.switchMode(mode)
        │                  → scope swap + Phoenix.rebirth
        ▼
new shell.initState() → consumePendingRoute()
        │                → drop if ts > 5 min old
        ▼
GoRouter.go(route) (customer) / Navigator.pushNamed(route) (seller)
```

### Why not move auth to Firebase too?

Supabase already owns the `users` and `seller_profiles` tables, and the Supabase JWT is what RLS keys off. Adding Firebase Auth would create two identities to keep in sync. FCM does not require Firebase Auth — only an FCM token, which is rotated and stored against `auth.uid()` in Supabase.

## 5. State Management — `flutter_bloc` (Cubits + BLoCs)

The codebase uses both `Cubit` and `Bloc` from `flutter_bloc ^9.0.0`. The rule of thumb is consistent across the code:

- **`Cubit`** when state transitions are simple direct calls (`load()`, `markRead()`, `toggleTheme()`).
- **`Bloc`** when state transitions are event-driven with multiple discrete events, and especially when events need transformers (`droppable`, `restartable`).

### Inventory

| Layer | Cubits | Blocs |
| --- | --- | --- |
| Root (cross-mode) | `ThemeCubit`, `AuthCubit`, `AppModeCubit`, `NetworkCubit`, `NotificationsCubit` | — |
| Customer mode | `ProfileCubit`, `ProfileOrdersCubit`, `ProductListCubit`, `CheckoutCubit` (extra step state) | `HomeBloc`, `CatalogBloc`, `CategoriesBloc`, `ProductDetailBloc`, `CartBloc`, `FavoritesBloc`, `CheckoutBloc`, `AddressesBloc`, `OrdersBloc`, `OrderDetailBloc` |
| Seller mode | `AddProductCubit`, `SellerDashboardCubit` (factory) | `DashboardBloc`, `SellerProductsBloc`, `ProductFormBloc`, `SellerOrdersBloc`, `SellerOrderDetailBloc`, `ShopSettingsBloc`, `ServicesBloc`, `TariffUpgradeBloc`, `VerificationBloc` |

### Sealed state classes

States are modelled as `sealed class` hierarchies with `Equatable` for value-equality, e.g. `AppAuthState` in `core/auth/auth_cubit.dart`:

```dart
sealed class AppAuthState extends Equatable { const AppAuthState(); }
class AppAuthUnauthenticated extends AppAuthState { ... }
class AppAuthAuthenticated extends AppAuthState {
  final String userId;
}
```

### Scoping rules

| Scope | Wired via | Lives… |
| --- | --- | --- |
| Root `BlocProvider.value` (in `main.dart`) | `ThemeCubit`, `AuthCubit`, `AppModeCubit` | Entire app lifetime |
| Mode singletons (in `service_locator`) | `HomeBloc`, `CartBloc`, `FavoritesBloc`, `CategoriesBloc`, `SellerDashboardCubit`, … | Until the next mode swap |
| Screen-local `BlocProvider(create: ...)` | `ProductListCubit`, `ProductDetailBloc`, `CheckoutBloc`, `AddProductCubit` | Until the route is popped |

### Realtime → BLoC bridge

Long-lived Supabase Realtime subscriptions live in **services** (not cubits) so they keep running across screen changes:

- `OrderTrackingService` (customer mode) — subscribes to `orders` rows where `user_id == auth.uid()`.
- `NewOrdersListener` (seller mode) — subscribes to `orders` rows where `shop_id == myShop.id`.
- `NotificationsCubit` (root scope) — subscribes to `public.notifications` filtered by user.

Services push state into cubits; cubits expose immutable snapshots to widgets.

## 6. Routing

### Customer mode — `go_router` (declarative)

`lib/customer/router.dart` builds a `GoRouter` with:

- `navigatorKey: customerNavigatorKey` (used by push handler to navigate from outside `context`).
- Two observers: `TalkerRouteObserver(talker)` (analytics-like) + `ConsoleNavObserver()`.
- A top-level `redirect` that gates `/tutorial`:

```dart
redirect: (context, state) {
  final atTutorial = state.matchedLocation == '/tutorial';
  if (!isTutorialSeen() && !atTutorial) return '/tutorial';
  if (isTutorialSeen() && atTutorial) return '/';
  return null;
},
```

Routes include:

```
/                       → CustomerHomeShell (bottom nav)
/tutorial               → CustomerTutorialScreen
/categories             → CategoriesScreen
/catalog?category=...   → CatalogScreen (query-param filters)
/product-list           → ProductListScreen (per-category)
/product-detail/:id     → SupabaseProductDetailScreen
/search                 → SearchScreen
/products/:slug         → SupabaseProductDetailScreen (slug-based)
/cart                   → CartScreen
/favorites              → FavoritesScreen
/checkout               → CheckoutScreen (extra: List<CartItemModel>)
/orders                 → OrdersHistoryScreen
/orders/:id             → OrderDetailScreen
/profile/addresses      → AddressesScreen
/seller/onboarding      → SellerOnboardingScreen
```

### Seller mode — traditional `Navigator` with `onGenerateRoute`

`lib/seller/seller_app.dart` is a plain `MaterialApp` with:

```dart
home: const SellerHomeShell(),
navigatorKey: sellerNavigatorKey,
onGenerateRoute: ...
```

This is intentional: seller flows are mostly modal (forms, sheets), and `go_router`'s declarative model added more friction than benefit during early seller-side development. It also pulls pending push routes via `_consumePendingRoute()` in `initState`.

> See `REFACTORING.md` §3 for the recommendation to migrate seller mode to `go_router` for parity.

### Auth guards

There are **no explicit route-level auth guards**. Instead:

- Customer profile screens render a guest CTA when `AuthCubit.state is AppAuthUnauthenticated`.
- The `AppModeCubit` demotes an unapproved seller back to customer mode at boot if the cached `seller_approved` flag is false.

## 7. Repository Layer — Multi-Backend Strategy

Every cross-mode repository follows the same pattern: **abstract interface → multiple implementations → DI picks one based on `AppConfig.useMocks` + Supabase availability**.

### Example: `ProductRepository`

```
ProductRepository (abstract)
├── MockProductRepository       — canned data with client-side filtering
├── RemoteProductRepository     — Dio-based; hits FastAPI / Edge Functions
└── SupabaseProductRepository   — direct Postgres via supabase_flutter
```

### Hybrid pattern — cart & favorites

`HybridCartRepository` and `HybridFavoritesRepository` exist because both need to work **before the user signs in** (anonymous cart) and **survive a network outage** (Hive cache).

```
HybridCartRepository
├── HiveCartRepository      — local cache (source of truth offline)
├── SupabaseCartRepository  — remote source of truth when authed
└── on auth state change:   — pushes local items to remote on sign-in,
                              falls back to local-only on sign-out
```

This is the only place where the codebase treats Hive as a peer of the backend rather than a passive cache.

| Domain | Interface | Implementations |
| --- | --- | --- |
| Products (read) | `ProductRepository` | Mock / Remote / Supabase |
| Products (data source) | `SupabaseProductDataSource` | Supabase only |
| Categories | `CategoryRepository` | Mock / Supabase |
| Shops | `ShopRepository` | Mock / Supabase |
| Banners | `BannerRepository` | Mock / Supabase |
| Cart | `CartRepository` | Mock / Remote / Hive / Supabase / Hybrid |
| Favorites | `FavoritesRepository` | Mock / Remote / Hive / Supabase / Hybrid |

## 8. Localization — Custom In-Dart System

`easy_localization` was removed and replaced by a hand-rolled system in `lib/core/i18n/`. Rationale (from `pubspec.yaml` comment): keeping translations as pure Dart constants allows Shorebird code-push to ship copy edits OTA without rebuilding the binary.

## 10. Native Bridges

### Yandex MapKit

`lib/core/maps/yandex_mapkit_initializer.dart` works around plugin lifecycle quirks:

- **Android** — the plugin's `onAttachedToActivity` starts `MapKitFactory.getInstance().onStart()` eagerly, which floods logcat with `SecurityException`s before location permission is granted. `MainActivity.kt` calls `onStop()` at boot; this Dart helper calls `onStart()` via a `MethodChannel('com.mebellar.app/yandex_mapkit')` *after* the permission prompt resolves.
- **iOS** — the plugin's `register(with:)` resolves `YMKMapKit.mapKit` immediately, which requires the API key to be set in `AppDelegate.swift` **before** `GeneratedPluginRegistrant`. No permission prompt is fired until a map asks for coordinates.

This is the only place in the codebase that uses a custom `MethodChannel`.

### Push notifications

The notification channel id `news` must match `AndroidManifest.xml`'s `com.google.firebase.messaging.default_notification_channel_id` meta-data and the channel pre-created by `flutter_local_notifications` during `PushService.bootstrap()`.
